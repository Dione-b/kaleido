# v1.0.0 Release Spec

## Motivation

Ship the first stable public Kaleido release with an explicit support contract for JS/TS teams building Soroban dApps. `v1.0.0` freezes the documented public workflow, package boundaries, and machine-readable error surface so consumers can automate around Kaleido without depending on internal implementation details.

This release exists to turn the current pre-v1 promise into an auditable contract:

- stable package names on npm;
- stable end-to-end workflow from project scaffold to contract invocation;
- stable client entrypoints for generated bindings, artifact lookup, XDR construction, and wallet signing;
- stable release gating for when `latest` is allowed versus when only `next` is allowed.

The release is public only when all five v1 viability specs are implemented and accepted, and when scheduled smoke evidence shows the supported flow works against testnet repeatedly rather than once.

## Non-goals

- Add new commands outside the documented public flow.
- Publish `latest` before all v1 readiness gates and smoke evidence are complete.
- Support parsing of human-readable error messages by automation.
- Promise stability for undocumented internals, private modules, or repository layout.
- Introduce multi-contract dependency deploy beyond the separate v1 viability spec that gates stable release readiness.
- Change the product from a thin orchestration layer over Stellar CLI, Stellar SDK, and generated bindings into a replacement for those tools.
- Store secret keys, seed phrases, or wallet secrets.

## Prior Art

Kaleido already documents a pre-v1 product promise in `README.md`: a predictable Soroban workflow built around `init -> build -> deploy -> generate -> invoke`, published packages `@kaleido/cli`, `@kaleido/core`, and `@kaleido/client`, and a pre-v1 release posture under `0.x` or `next`.

`docs/release/v0.1.0-alpha.md` established the first explicit release document for an internal alpha milestone. That alpha proved the initial workflow and client interop layer, but it intentionally stopped short of an npm stability guarantee and did not define a canonical public `latest` release contract.

`docs/release/v1-readiness.md` and `docs/superpowers/specs/00-v1-viability-index.md` already define the blocking release gates. This spec is the public contract layered on top of those gates: it states what becomes stable at `v1.0.0`, what remains out of scope, and what evidence must exist before the stable dist-tag is unfrozen.

Rejected alternatives:

- Keep relying on README language alone. Rejected because a stable release needs one canonical contract with rollout, rollback, ownership, evidence, and machine-facing guarantees in one place.
- Publish `latest` as soon as packages can be built and pushed. Rejected because build success alone does not prove consumer isolation, error contract completeness, or repeated scheduled smoke success.
- Treat `next` and `latest` as equivalent once package READMEs are ready. Rejected because the repo already distinguishes pre-v1 preview distribution from stable public support.

## Interface Contract

### Published Packages

- `@kaleido/cli`
- `@kaleido/core`
- `@kaleido/client`

`@kaleido/core` is a published package but is not the primary end-user entrypoint. The stable consumer-facing surfaces are the CLI and documented client APIs. Anything not documented as public API remains outside the v1 support contract even if it ships in a published package.

### Supported CLI Flow

`init -> build -> deploy -> generate -> invoke`

The v1 release guarantees this flow as the supported happy path for a Soroban project using Kaleido conventions. Each command must remain callable through `@kaleido/cli` on supported platforms, and the output of one step must remain consumable by the next without requiring manual edits to contract IDs or manual SCVal serialization.

### Supported Client Surface

Supported root exports from `@kaleido/client`:

- `createKaleidoClient`
- `resolveContractId`
- `createDefaultBindingAdapter`
- `KaleidoContractClient`
- `buildXdr`
- `KaleidoBindingAdapter`
- `KaleidoClientConfig`
- `KaleidoContractRegistration`
- `KaleidoInvokeOptions`
- `KaleidoInvokeResult`
- `KaleidoInvokeStatus`
- `KaleidoNetwork`
- `KaleidoWalletAdapter`
- `KaleidoXdrBuildResult`

Supported subpath exports:

- `@kaleido/client/freighter` -> `freighterWalletAdapter`

Primary documented browser/client flow:

- `createKaleidoClient`
- `client.contract(name).invoke(method)`
- `client.contract(name).buildXdr(method)`
- `@kaleido/client/freighter` -> `freighterWalletAdapter`

The primary documented flow is the default browser integration path:

- `createKaleidoClient` wires network config, artifacts, generated bindings, and wallet adapter into one client instance.
- `client.contract(name).invoke(method)` prepares and submits an invocation through the registered binding and wallet flow.
- `client.contract(name).buildXdr(method)` returns the invocation XDR construction path without requiring submission.
- `freighterWalletAdapter` from `@kaleido/client/freighter` is the supported wallet adapter for the v1 browser/client flow.

The remaining root exports above are also public and supported for consumers that need lower-level composition or explicit TypeScript integration.

- `resolveContractId` remains a supported public helper for contract ID resolution from Kaleido artifacts and explicit registration data.
- `createDefaultBindingAdapter` remains a supported public helper for consumers that need direct binding adapter composition.
- `KaleidoContractClient` remains a supported public class export for consumers that need lower-level contract client control than `createKaleidoClient`.
- `buildXdr` remains a supported public root helper for direct XDR preparation flows outside the higher-level client facade.

Undocumented helpers, private module paths, and implementation details behind these exports are not part of the v1 public contract.

### Public Error Contract

Automation may parse `KALEIDO_*` codes and must not parse human-readable messages.

The v1 contract requires:

- all public failures emitted by Kaleido surfaces to carry a documented `KALEIDO_*` code;
- human-readable text to remain descriptive for operators but non-normative for automation;
- additions of new codes to follow documentation and release-note updates;
- removals or semantic changes to documented codes to be treated as breaking changes.

### Release SLA

`v1.0.0` does not promise a latency or uptime SLA for public blockchain infrastructure. The release SLA is a compatibility and correctness SLA:

- the documented CLI flow must remain functional on supported toolchain versions;
- the documented client surface must remain callable without undocumented integration steps;
- `latest` must not be published unless scheduled smoke evidence demonstrates repeated success on the supported flow.

## Compatibility Contract

The v1 compatibility promise is scoped to the documented public surface, not every repository internal.

- Node.js: `>=20`
- pnpm: `>=9`
- Stellar CLI: constrained by the separate Stellar CLI version contract and its supported range
- Package versioning: breaking changes to documented public surfaces require a new major version after `v1.0.0`
- Dist-tags: pre-v1 preview distribution may use `0.x`, `alpha`, `beta`, or `next`; stable consumers must use `latest` only after readiness gates are satisfied

Compatibility rules:

- Generated projects and consumer installs must work without importing private package paths.
- `kaleido.artifacts.json` and `kaleido.config.ts` behavior relied on by the documented flow must remain compatible for the supported commands.
- Secret-key-based flows remain unsupported; `--source` continues to mean identity alias or public `G...` address only.
- A change that breaks automation built on documented `KALEIDO_*` codes, CLI flow order, or documented client entrypoints is a breaking change.

## Observability Plan

The release owner must be able to answer two operational questions before `latest` is published:

1. Does the documented workflow pass in deterministic local/CI checks?
2. Does the same workflow succeed repeatedly in scheduled live testnet smoke runs?

Required observability signals:

- `pnpm typecheck`
- `pnpm build`
- `pnpm test`
- `pnpm test:consumer`
- scheduled `Testnet Smoke` workflow history in GitHub Actions
- documented `KALEIDO_*` error codes for public failure classification

Required operator checks:

- confirm the three newest scheduled `Testnet Smoke` runs are green;
- confirm there is no unretried hard smoke failure in the previous 7 days;
- confirm the smoke workflow validates the supported public flow rather than an internal-only path.

This release does not add product telemetry. Observability comes from deterministic CI, consumer isolation tests, scheduled smoke evidence, and the documented error contract.

## Rollout

`v1.0.0` rollout is a gated publish, not an aspirational milestone. All steps below are required in order:

1. Confirm all five v1 viability specs are implemented and accepted.
2. Confirm `docs/release/v1-readiness.md` is fully satisfied.
3. Run `pnpm typecheck`, `pnpm build`, `pnpm test`, and `pnpm test:consumer`.
4. Verify three consecutive successful scheduled `Testnet Smoke` runs and no unretried smoke failure in the previous 7 days.
5. Publish packages only after the above evidence is recorded.
6. Move the stable release to `latest` only after all evidence is complete.

### Dist-tag policy

- `next`: allowed once publish workflow, package READMEs, and consumer-isolation docs are complete.
- `latest`: blocked until all v1 readiness requirements and scheduled smoke evidence are complete.

## Rollback

Rollback rules depend on when the issue is discovered.

Before `latest` is published:

- stop the release;
- keep distribution on `next` only;
- fix the blocking issue or explicitly reduce scope and re-approve the release contract;
- rerun all required readiness and smoke verification.

After `latest` is published:

- do not rewrite or delete the public `v1.0.0` tag;
- do not silently change the meaning of the documented v1 contract;
- cut a follow-up patch release with the fix, or deprecate the affected package/version if necessary;
- document the issue, impact, and recovery path in release notes or follow-up release documentation.
